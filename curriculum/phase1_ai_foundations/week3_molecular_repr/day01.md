# Day 1: SMILES와 분자 문자열 표현

> 이전 학습(Week 2 Day 5)에서 AI 신약개발 회사 지도를 그리며 각 회사의 "핵심 기술" 칸을 채우려면 분자 표현(molecular representation)을 이해해야 한다고 예고했습니다. 오늘부터 Week 3은 한 단계 아래로 내려가, **컴퓨터가 화학을 이해하는 법**을 다룹니다. 첫날인 오늘은 그 출발점이자 모든 화학 데이터베이스의 공용어인 **SMILES**를 학습합니다.

## 개요

SMILES(Simplified Molecular Input Line Entry System, 단순화 분자 선형 입력 표기)는 분자 구조를 한 줄의 ASCII 텍스트 문자열로 표현하는 방식입니다. 1988년 David Weininger가 제안한 이래, 거의 모든 화학정보학(cheminformatics) 도구와 데이터베이스(ChEMBL, PubChem, DrugBank)가 분자를 저장·교환하는 표준이 되었습니다. AI 신약개발의 관점에서 SMILES가 중요한 이유는 명확합니다 — **분자를 텍스트로 바꾸면, 자연어 처리(NLP)를 위해 만들어진 거대한 AI 기술(RNN, Transformer, 언어모델)을 분자에 그대로 적용할 수 있기** 때문입니다. 오늘은 SMILES의 문법, 한 분자가 여러 문자열로 표현되는 비유일성 문제와 그 해결책(정규화), 그리고 약학 전공자가 반드시 챙겨야 할 입체화학·이온화 정보가 SMILES에 어떻게 담기는지를 다룹니다.

## 핵심 개념

### SMILES란 무엇인가 — 분자를 한 줄의 텍스트로

SMILES는 분자 그래프(원자=노드, 결합=엣지)를 **깊이 우선 탐색(depth-first traversal)으로 한 번 훑으면서** 마주치는 원자와 결합을 문자로 기록한 결과물입니다. 핵심 표기 규칙은 다음과 같습니다.

| 요소 | 표기 | 예시 |
|------|------|------|
| **원자** | 원소 기호. 유기 원자(C, N, O, S, P, F, Cl, Br, I)는 대괄호 생략 가능 | `C`(탄소), `O`(산소), `[Na+]`(그 외는 대괄호) |
| **결합** | 단일=생략, 이중=`=`, 삼중=`#` | `CC`(에테인), `C=C`(에텐), `C#N`(나이트릴) |
| **분지(branch)** | 괄호로 곁가지 표현 | `CC(=O)O`(아세트산) |
| **고리(ring)** | 같은 숫자로 고리가 닫히는 두 원자를 표시 | `C1CCCCC1`(사이클로헥세인) |
| **방향족(aromatic)** | 소문자로 표기 | `c1ccccc1`(벤젠) |
| **수소** | 보통 원자가(valence)로 자동 채움(암묵적 수소) | `O`는 자동으로 H₂O가 아니라 물 분자의 산소로 해석 |

예를 들어 **아스피린**은 `CC(=O)Oc1ccccc1C(=O)O`로 표현됩니다. 약학 전공자라면 이 문자열에서 아세틸기(`CC(=O)O`), 방향족 고리(`c1ccccc1`), 카복실산(`C(=O)O`)을 즉시 분해해 읽을 수 있습니다. 이것이 오늘의 첫 번째 실용 기술입니다 — **SMILES를 작용기 단위로 해독하는 능력**.

### 비유일성과 정규화(canonicalization)

SMILES의 가장 중요한 함정은 **하나의 분자가 여러 개의 올바른 SMILES로 표현될 수 있다**는 점입니다. 그래프를 어느 원자에서 출발해 훑느냐에 따라 문자열이 달라지기 때문입니다. 에탄올은 `CCO`, `OCC`, `C(O)C`가 모두 같은 분자입니다.

이 문제는 데이터에 치명적입니다. 같은 약물이 데이터베이스에 서로 다른 SMILES로 중복 저장되면, AI 모델은 이를 다른 분자로 오인합니다. 해결책이 **정규 SMILES(canonical SMILES)** 로, 각 분자에 대해 항상 동일한 하나의 문자열을 생성하는 알고리즘입니다(Morgan 알고리즘 기반의 원자 순위 매기기 사용). RDKit 같은 도구는 입력 SMILES를 정규형으로 변환해 중복을 제거합니다.

> 핵심: 데이터를 다룰 때 **반드시 정규 SMILES로 표준화**해야 합니다. 이는 Phase 1 Week 4(데이터 품질)에서 다룰 "쓰레기를 넣으면 쓰레기가 나온다"의 첫 사례입니다.

### 약학 지식과의 대응 — 입체화학·토토머·이온화

여기서 약학 전공자의 경쟁력이 직접 발현됩니다. SMILES가 분자를 텍스트로 압축할 때 **임상적으로 결정적인 정보가 누락되거나 왜곡될 수 있고, 그것이 무엇인지 판단하려면 약학 지식이 필요**합니다.

- **입체화학(stereochemistry)**: SMILES는 카이랄 중심을 `@`/`@@`로, 이중결합 기하를 `/`·`\`로 표기합니다. 약학 전공자는 거울상 이성질체(enantiomer)가 약리학적으로 전혀 다를 수 있음을 압니다 — 탈리도마이드(thalidomide)의 한 거울상은 진정 효과를, 다른 거울상은 최기형성을 보인 것이 대표적입니다(체내 라세미화 문제는 별개로 하더라도, 이성질체 구분의 임상적 중요성을 보여줍니다). 따라서 입체 정보가 없는 평면 SMILES로 학습한 ADMET 모델은 (R)/(S)-약물의 차이를 구분하지 못합니다. 비전공 개발자는 "SMILES가 다 같은 분자"라고 넘기지만, 약학 전공자는 **언제 입체 정보를 보존해야 하는지를 판단**할 수 있습니다.
- **토토머(tautomer)와 이온화 상태**: 같은 분자가 케토-엔올 토토머처럼 여러 형태로 존재하거나, 생리적 pH(7.4)에서 이온화(pKa에 따라)되어 다른 SMILES로 적힐 수 있습니다. 흡수·분포(Week 2의 ADME)는 이온화 상태에 좌우되므로, 어떤 형태를 "대표 구조"로 표준화할지는 약학적 판단입니다.
- **염(salt)과 혼합물**: 의약품은 흔히 염 형태(예: `...hydrochloride`)로 SMILES에 점(`.`)으로 연결된 여러 조각으로 저장됩니다. 활성 성분만 남기는 탈염(desalting)은 일상적인 전처리 단계입니다.

### SMILES의 한계를 메우는 표기 — SELFIES, InChI

SMILES는 생성 AI에서 약점을 드러냅니다. 문자 하나만 잘못 바꿔도(예: 고리 닫힘 숫자 불일치) **문법적으로 무효한 문자열**이 되기 때문입니다. 이를 해결하려 2020년 제안된 **SELFIES(Self-Referencing Embedded Strings)** 는 어떤 문자열이든 항상 유효한 분자로 해석되도록 설계되어, 생성모델(Phase 2에서 학습)의 출력 안정성을 크게 높였습니다. 한편 **InChI(IUPAC International Chemical Identifier)** 는 IUPAC가 표준화한 계층적 식별자로, 동일 분자에 항상 같은 식별자를 부여해 데이터베이스 간 매칭에 쓰입니다(해시값인 InChIKey 형태로 검색에 활용).

## 작동 원리와 아키텍처

SMILES가 AI 파이프라인에서 처리되는 흐름은 **입력(문자열) → 파싱(구조 복원) → 변환(모델 입력)** 의 3단계입니다.

```
[SMILES 처리 파이프라인]
1. 입력      : "CC(=O)Oc1ccccc1C(=O)O" (아스피린 SMILES 텍스트)
2. 파서      : RDKit MolFromSmiles → 분자 객체(Mol), 유효성 검사
3. 표준화    : 정규 SMILES 변환 + 탈염 + 토토머/이온화 정규화
4. 모델 입력 : 두 갈래로 분기
   - (a) 텍스트 경로: 토크나이저 → 토큰 ID → 임베딩 → 시퀀스 모델(RNN/Transformer)
   - (b) 구조 경로 : Mol → 분자 지문(Day 2) 또는 분자 그래프(Day 3)
```

(a) 경로가 오늘의 핵심입니다. SMILES를 **언어처럼 다루는** 것입니다. 토크나이저는 문자열을 의미 단위(원자·결합·괄호)로 쪼개고, 각 토큰을 벡터로 임베딩한 뒤, 자연어 모델과 똑같은 구조로 다음 토큰을 예측하도록 학습시킵니다. 이것이 "분자의 언어모델(chemical language model)"의 원리이며, Phase 2의 분자 생성과 Phase 3의 화학 Foundation Model로 곧장 이어집니다.

바이브코딩으로 이런 시스템을 만들 때의 설계 결정은 다음과 같습니다.

| 설계 결정 | 선택지 | 추천 | 이유 |
|-----------|--------|------|------|
| 표기 방식 | SMILES / SELFIES / InChI | 용도별 | DB 저장·표시는 정규 SMILES, 생성모델은 SELFIES, 식별·중복제거는 InChIKey |
| 파서 | RDKit / OpenBabel | RDKit | 파이썬 생태계 표준, 표준화·지문·그래프 변환 일괄 지원 |
| 토큰화 | 문자 단위 / 원자 단위(정규식) | 원자 단위 | `Cl`을 `C`+`l`로 쪼개는 오류 방지 |
| 입체화학 | 보존 / 제거 | 과제 의존 | ADMET·활성 예측은 보존, 단순 분류는 제거 가능 |

## 신약개발 적용

### 모든 화학 데이터의 저장 형식

SMILES는 추상적 개념이 아니라, 학습자가 앞으로 매일 다룰 데이터의 실제 형식입니다.

- **ChEMBL**: 생리활성 분자 약 240만 종(ChEMBL 34, 2024년 기준)을 SMILES로 저장하며, AI 신약개발 모델 학습의 표준 출처입니다(Phase 1 Week 4에서 심화).
- **PubChem**: 1억 종 이상의 화합물을 SMILES/InChI로 보유합니다.
- **초대규모 가상 라이브러리**: Enamine REAL Space는 합성 가능한 분자를 SMILES로 열거(enumerate)하여 2024년 기준 수백억 종 규모로 확장된 것으로 보도됩니다(정확한 수치는 출처 확인 필요). 이런 라이브러리 전체가 텍스트 문자열로 표현되기에 컴퓨터로 다룰 수 있습니다.

### 생성 AI의 첫 성공 — SMILES 기반 RNN

Segler 등은 2018년(*ACS Central Science*) SMILES를 학습한 순환신경망(RNN)으로 신약 후보 분자 라이브러리를 생성하는 연구를 발표했습니다. 이는 "분자를 언어로 다루면 새 분자를 '문장 쓰듯' 생성할 수 있다"는 패러다임의 출발점이 되었고, Insilico Medicine의 분자 생성 엔진 Chemistry42를 비롯한 상용 플랫폼이 이 계보 위에서 발전했습니다(Week 2 Day 5에서 다룬 INS018_055도 이런 생성·최적화 파이프라인의 산물입니다).

### 표기 방식 비교

| 항목 | SMILES | SELFIES | InChI |
|------|--------|---------|-------|
| 제안 | 1988(Weininger) | 2020(Krenn 등) | IUPAC |
| 사람 가독성 | 높음 | 낮음 | 낮음 |
| 유일성 | 정규화 필요 | 정규화 필요 | 본질적 유일 |
| 생성모델 적합성 | 무효 문자열 발생 | 항상 유효 | 부적합 |
| 주 용도 | 저장·표시·입력 | 생성모델 출력 | 식별·중복제거 |

## 창업 관점

Phase 1 단계이므로 좌표만 짚습니다. SMILES 표준화(정규화·탈염·입체/토토머 처리)는 모든 AI 신약개발 제품의 데이터 입력 계층(ingestion layer)에서 반드시 거치는 단계이며, **여기서의 약학적 판단(어떤 이온화·입체 형태를 대표로 쓸지)이 모델 데이터 품질의 차별점**이 됩니다. 구체적 데이터 파이프라인 제품화는 Phase 4에서 설계합니다.

## 오늘의 과제

1. **SMILES 해독·작성 실습 (30-40분)**: DrugBank 또는 PubChem에서 익숙한 약물 3개(예: ibuprofen, caffeine, atorvastatin)의 SMILES를 찾아, 각 문자열을 작용기 단위(고리·카복실산·아마이드 등)로 분해해 손으로 주석을 답니다. 약학 지식으로 어떤 작용기가 약리 활성·대사와 관련되는지 1줄씩 메모합니다.
2. **정규화·입체화학 확인 (40분)**: 같은 분자의 SMILES를 2~3가지 방식으로 써보고(예: 에탄올 `CCO`/`OCC`), RDKit 온라인 도구나 PubChem의 구조 검색으로 모두 동일 분자임을 확인합니다. 이어 카이랄 약물(예: ibuprofen의 (S)-체) SMILES에서 `@`/`@@`가 어디에 붙는지 찾고, 입체 정보를 제거하면 무엇이 사라지는지 3~4줄로 정리합니다.
3. **SELFIES 핵심 파악 (30분)**: Krenn 등(2020)의 SELFIES 논문 Abstract를 읽고, "SMILES의 어떤 문제를 어떻게 해결했는가"를 핵심 3줄로 요약합니다. 이것이 Phase 2 생성모델에서 왜 중요해질지 1문장으로 예측해 둡니다.

## 참고 자료

- Weininger, D. (1988). "SMILES, a chemical language and information system. 1. Introduction to methodology and encoding rules." *Journal of Chemical Information and Computer Sciences*, 28(1), 31-36. — SMILES의 원전. 표기 규칙의 표준 정의(역사적 기초 문헌).
- Krenn, M. *et al.* (2020). "Self-referencing embedded strings (SELFIES): A 100% robust molecular string representation." *Machine Learning: Science and Technology*, 1(4). — SMILES의 유효성 문제를 해결한 최신 표기. 생성모델의 기반.
- Segler, M.H.S. *et al.* (2018). "Generating focused molecule libraries for drug discovery with recurrent neural networks." *ACS Central Science*, 4(1), 120-131. — SMILES 기반 분자 생성의 출발점.
- RDKit (rdkit.org) — SMILES 파싱·정규화·변환의 사실상 표준 오픈소스 라이브러리.
- ChEMBL (ebi.ac.uk/chembl) — SMILES로 저장된 생리활성 분자 데이터베이스. AI 모델 학습의 핵심 출처.
